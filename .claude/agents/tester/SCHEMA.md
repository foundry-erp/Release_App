# TESTER — SCHEMA

## Files Tester Reads

### Phase-Level Testing
- {phase}/validator/validated.md (spec — exact test_commands and pass_conditions)
- {phase}/builder/built.md (contract — what was built and how to reach it)
- src/** (source files — read-only for understanding)
- tests/** (test files — execute these)

### Final Integration Testing
- All phases' validated.md files (for cross-phase flows)
- All phases' built.md files (for HOW_TO_RUN aggregation)
- PHASE_INDEX.md (to know which phases are complete)

## Required Sections in test-report.md

All sections in the template are REQUIRED.
Missing any section = testing incomplete = schema failure.

## Special Section Rules

### Execution Summary Table
- Must include all four layers: unit, integration, e2e, performance
- status: ✅ PASS | ❌ FAIL | ⏭ SKIPPED
- SKIPPED only when earlier P0 failure triggered early exit
- time: actual execution time (not estimated)

### Passed Tests Section
- One line per passing test
- No detail (passing tests generate no noise)
- Format: "✅ {test description}"

### Failed Tests Section
- One FAIL-{ID} block per failure
- All seven fields required: test, expected, actual, criterion_violated, likely_file, failure_type, severity
- failure_type must be one of: DETERMINISTIC | SPEC_GAP | ENVIRONMENTAL | FLAKY
- severity: P0 (blocking) | P1 (non-blocking)

### FAILURE_TYPE Field (Top Level)
- Set when OVERALL_STATUS = FAIL
- Must be: DETERMINISTIC | SPEC_GAP | ENVIRONMENTAL | FLAKY
- Set to "none" when OVERALL_STATUS = PASS
- If multiple failures with different types: use the most common type

### Acceptance Criteria Coverage Table
- Must list every AC from validated.md
- status: ✅ PASS | ❌ FAIL
- verified_by: which test(s) covered this AC

### HOW_TO_RUN Section
- Exact commands to start the application from scratch
- Exact commands to run tests
- Must be copy-pasteable (no placeholders)

### MANUAL_TESTS Section
- Step-by-step instructions for human verification
- Each step: action → expected result
- Must be specific enough for non-technical user

## Failure Type Identification Guide

| Signal | Type |
|--------|------|
| Same error, points to specific line, reproducible | DETERMINISTIC |
| No wrong line — spec just didn't cover this case | SPEC_GAP |
| Error fires before application code starts | ENVIRONMENTAL |
| Passed with identical code in a previous run | FLAKY |

## Scaffolded Tests Format

When a test layer has no test files:

```markdown
## Scaffolded Tests
- SCAFFOLDED: tests/unit/EmailService.test.ts
  AC-3.1 "Email validation must reject invalid formats" had no test coverage.
  Created minimal test using test_command: npm test -- EmailService

- SCAFFOLDED: tests/integration/auth-flow.test.ts
  AC-1.4 "Login flow must set session cookie" had no test coverage.
  Created minimal test from validated.md Manual Test Steps.
```

Scaffolding is minimal and functional — not comprehensive.
Builder should create proper tests, but Tester scaffolds when missing.
